# Solana 链上「按程序找交易」——通俗流程说明

这份文档用**大白话**解释：为什么在 Solana 上做「某类合约（program）在一段时间/一段区块里干了什么」会比你在以太坊上习惯的玩法**更显麻烦**；以及我们这套脚本**到底分几步、每步在干什么**。

---

## 1. 先用一个比喻

- **以太坊**：你手里已经有很多「书单号」（交易 hash），去柜台 `查这本书` → 一次一本，很清楚。  
  若要知道「这个柜架（合约）最近被谁借过」，有时还能用 **日志 / 事件** 直接筛，不必把整座图书馆逐格翻一遍。

- **Solana**：**查某一本书**也可以说——只要你有**这本书的书签号**（交易的 **signature**），RPC 一样能 `getTransaction` 把内容调出来。  
  **难的是**：没人给你一张现成清单，写着「在过去 1 万个区块里，所有动过某某 program 的书签号」。公共 RPC **默认不提供**这种「按 program + 按时间段一次性列清单」的便宜接口。

所以：  
**不是「Solana 查一笔交易特别难」**，而是**「先弄清楚要查哪几笔」**在公共 RPC 上往往要你自己多干活。

---

## 2. 我们这套脚本在干什么？（六步）

把它想成：**先列名单 → 再按名单把正文拉下来 → 再整理成表格、写报告**。

| 步骤 | 名字 | 白话 |
|------|------|------|
| ① | 发现 program | 从网上和链上线索里**搜罗**「可能是 PropAMM 相关的程序地址」，合并成一张**待观察名单**（`programs.yaml`）。 |
| ② | 合并配置 | 把「自动搜到的」和「你手抄进文件的」合并成**最终名单**。 |
| ③ | **收集签名**（二选一） | **目的**：得到一张「有意思的交易」的 **signature 列表**（存在 SQLite 里）。 |
| | **方式 A：按 program 翻历史** | 像翻电话簿：从**最新一页**往前翻（`getSignaturesForAddress`），只保留落在你要的**日期**里的号码。 |
| | **方式 B：按区块一段段扫** | 从你给的 **slot 起止** 一格一格看（`getBlock`），看这一格里有没有**调用名单里某个 program**，有就把这笔交易的 signature 记下来。 |
| ④ | 拉全文明文 | 对名单里每一笔 signature 调 **`getTransaction`**，把完整 JSON 存到硬盘（和以太坊「有 hash 再调一次 RPC」是同类事）。**条数一多，这一步就会**特别久**。 |
| ⑤ | 解析 | 读存下来的 JSON，回答一些简单问题：有没有经过 Jupiter、大概涉及哪些 program 等。 |
| ⑥ | 出报表 | 生成 `summary.md` 之类，方便交作业或给导师看。 |

Docker 里跑 `collector`，就是按顺序自动做完 ①～⑥（也可以通过环境变量**跳过**某几步，详见 `README.md`）。

---

## 3. 为啥「扫块」会又慢又占空间？

- **扫块**：区间里有 **N 个 slot**，理想情况就要 **N 次 `getBlock`**（像把 N 个抽屉各打开看一遍）。  
- **1 万个 slot** 就是约 **1 万次请求**，这只是**列名单**；若筛出来几十万笔「相关」交易，后面还要**几十万次 `getTransaction`**，磁盘上也会存**大量 JSON 文件**。

所以它不是 bug，是**工作量本来就大**；换再贵的节点，也只是「每秒能多问几句」，不能变成「一次 RPC 返回整段历史全部相关交易」（除非用**索引/增强 API**）。

---

## 4. 和你在以太坊上做套利检测的对比（一句话）

| | 以太坊（你熟悉的） | Solana（我们这次） |
|---|-------------------|-------------------|
| **已经知道 tx 标识** | `eth_getTransaction*` 按 hash 拉 | `getTransaction` 按 signature 拉 → **一样顺手** |
| **还没名单，要从链上「挖」** | 常靠合约事件、日志、或自己扫块策略 | 公共 RPC 缺「按 program+区间一键列签名」→ 往往要 **翻签名页** 或 **逐块扫** |

---

## 5. 想少折腾时可以怎么用这份脚本？

- **名单别太胖**：扫块时用 `PROGRAMS_CONFIG` 指向**只含目标 PropAMM**的 yaml，避免把「超级大热门 DEX」也放进名单，否则**几乎每笔都命中**，SQLite 爆炸。  
- **先少拉正文**：设 **`FETCH_LIMIT=5000`** 之类，先验证解析和报表，再决定要不要全量 `getTransaction`。  
- **干脆不拉正文**：只保留 `SKIP_FETCH=1`，只用签名列表做统计（适合只要「有多少笔」的场景）。

## 6. 一张极简流向图

```mermaid
flowchart LR
  A[① 找 program 地址] --> B[② 合成 programs.yaml]
  B --> C[③ 列 signature 名单]
  C --> D[④ 每笔 getTransaction 存盘]
  D --> E[⑤ 解析 JSON]
  E --> F[⑥ 报表]
```

**③** 两条路：**翻 program 历史** 或 **按 slot 扫块**——都是在解决同一件事：**在没有「官方一键列表」的前提下，自己造出要写进数据库的 signature 名单**。

---

若你只记一句：**Solana 上「查某一笔」不难，难的是「先弄清用 RPC 在大量区块里该查哪几笔」；我们的脚本是把这个笨活拆成可重复跑的流水线。** 更细的参数见项目里的 `README.md`。
